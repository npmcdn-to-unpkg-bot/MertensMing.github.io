Redux 学习笔记

发表于 2016-08-22

为什么需要 Redux？

去实习做了项目，感触还挺深刻的，项目中需要处理很多事件，触发事件的变化就会使应用的状态（state）变化，而且有时候使得 state 变化的原因不仅仅是触发事件，也有可能是服务器响应、缓存数据、激活的路由 …等各种各样的原因。

如果一个 model 的变化会引起另一个 model 变化，那么当 view 变化时，就可能引起对应 model 以及另一个 model 的变化，依次地，可能会引起另一个 view 的变化。直至你搞不清楚到底发生了什么。state 在什么时候，由于什么原因，如何变化已然不受控制。 当系统变得错综复杂的时候，想重现问题或者添加新功能就会变得举步维艰。
如果这还不够糟糕，考虑一些来自前端开发领域的新需求，如更新调优、服务端渲染、路由跳转前请求数据等等。

—— 《Redux 文档》

上面提到的复杂性很大程度来自于： 我们总是将两个难以厘清的概念混淆在一起：变化和异步。

这句话深有感触，同意 +1！！！！有上面那么多东西，写代码的时候已经足够让人炸裂了。

Redux 的三大原则

单一数据源

整个应用的 state 被存放在一个 对象(object) 里面，而这个对象被只存在于唯一的一个 store 中。就是说整个应用的 state 都需要通过这个 store 来管理。

只读的 state

惟一改变 state 的方法就是触发 action，action 是一个用于描述已发生事件的普通对象。需要注意的就是，你不能去修改 state，要想改变 state，只能返回一个新的 state，而不能在原来的基础上修改。

使用纯函数（Reducer）改变 state

为了描述 action 如何改变 state 的 ，需要编写 reducers。Reducer 只是一些纯函数，它接收先前的 state 和 action，并返回新的 state。
ps：不要把 reducer 想象得太复杂，其实就是一个函数，接收 state 和 action，根据 action 返回新的 state 或者旧的 state。

一些基础知识

action

简单的来说，action 就是一个对象

  1. 这个对象必须包含 type 字段，通常都是大写字母的形式
  2. 其他属性你可以随便定义
  3. 但是！这里有个参考，让你定义标准的 action，当然，你也可以不去理会。
  4. 可以通过 store.dispatch() 将 action 传到 store。

action creator

顾名思义，action creator 就是生成 action 的方法。

发送 action

  1. redux 只需要把 action creator 的结果传递给 dispatch 方法即可发起一次 dispatch 过程。
  2. 或者创建一个被绑定的 action creator 来自动 dispatch
  3. 使用 bindActionCreators(actionCreators, dispatch) 方法来绑定 action，最后调用的时候直接 dispatch

bindActionCreators

bindActionCreators 做了两件事：

  1. 将 action creator 用 dispatch 包起来，方便以后直接调用他们
  2. 将被 dispatch 包起来的 action creator 存放进一个对象里面，key 就是 action creator 的名字

那么问题来了，dispatch 方法从哪里来呢？这个是比较重要的，我们需要记住 dispatch 有两个来源：

  1. store 的实例方法
  2. react-redux 也会提供 dispatch

继续看回 bindActionCreators 方法，这个方法接受两个参数

Reducer

处理 action

reducer 就是一个纯函数，接收旧的 state 和 action，返回新的 state。

保持 reducer 纯净

永远不要在 reducer 里做这些操作：

  • 修改传入参数；
  • 执行有副作用的操作，如 API 请求和路由跳转；
  • 调用非纯函数，如 Date.now() 或 Math.random()。

只要传入参数相同，返回计算得到的下一个 state 就一定相同。
没有特殊情况、没有副作用，没有 API 请求、没有变量修改，单纯执行计算。

我们来看看这个 combineReducers 函数的本质，它主要也就是做了下面 3 件事：

  1. 接收一个对象
  2. 生成一个函数
  3. 这个函数返回一个对象

store

store 的职责

  • 维持应用的 state
  • 提供 getState() 方法获取 state
  • 提供 dispatch(action) 方法更新 state
  • 通过 sbuscribe(listener) 注册监听器
  • 通过 subscribe(listener) 返回的函数注销监听器

再次强调 redux 应用只有一个 store，当需要拆分数据处理逻辑的时候，应该使用 reducer 组合而不是创建多个 store。

创建 store

  1. 使用 combineReducers() 将多个 reducers 合并成一个
  2. 将其导入，传递给 createStore()

createStore() 的第二个参数是可选的, 用于设置 state 初始状态

发起 action

数据流

严格的单向数据流是 Redux 架构的核心。

redux 应用中数据的声明周期遵循下面四个步骤：

(1) 调用 store.dispatch(action)

action 就是一个描述 “发生了什么” 的对象

可以在任何地方调用 store.dispatch(action)

(2) store 调用传入的 reducer 函数

Store 会把两个参数传入 reducer： 当前的 state 树和 action。

(3) 根 reducer 应该把多个子 reducer 输出合并成一个单一的 state 树

  1. 根 reducer 的结构完全由你决定。
  2. Redux 原生提供combineReducers()辅助函数，来把根 reducer 拆分成多个函数，用于分别处理 state 树的一个分支。

搭配 React

安装 react-redux

容器组件和展示组件相分离

Redux 的 React 绑定库包含了 容器组件和展示组件相分离 的开发思想。

  • 明智的做法是只在最顶层组件（如路由操作）里使用 Redux。
  • 其余内部组件仅仅是展示性的，所有数据都通过 props 传入。
容器组件 展示组件
Location 最顶层，路由处理 中间和子组件
Aware of Redux 是 否
读取数据 从 Redux 获取 state 从 props 获取数据
修改数据 向 Redux 派发 actions 从 props 调用回调函数

连接到 Redux

我们需要做出两个变化，将 App 组件连接到 Redux 并且让它能够 dispatch actions 以及从 Redux store 读取到 state。

Provider

首先，我们需要获取从之前安装好的 react-redux 提供的 Provider，并且在渲染之前将根组件包装进 <Provider>。

这使得我们的 store 能为下面的组件所用。

connect()

接着，我们想要通过 react-redux 提供的 connect() 方法将包装好的组件连接到Redux。
尽量只做一个顶层的组件，或者 route 处理。从技术上来说你可以将应用中的任何一个组件 connect() 到 Redux store 中，但尽量避免这么做，因为这个数据流很难追踪。

任何一个从 connect() 包装好的组件都可以得到一个 dispatch 方法作为组件的 props，以及得到全局 state 中所需的任何内容。

connect() 的唯一参数是 selector。此方法可以从 Redux store 接收到全局的 state，然后返回组件中需要的 props。
最简单的情况下，可以返回一个初始的 state （例如，返回认证方法），但最好先将其进行转化。
