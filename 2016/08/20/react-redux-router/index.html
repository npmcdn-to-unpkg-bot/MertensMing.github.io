使用 React、React-router 和 Redux 开发前端应用

发表于 2016-08-20   |   分类于 JavaScript

前言

这篇笔记是接着上一篇的，使用的是同一个示例，文章主要介绍一个 web 应用是如何跟 redux 和 react-router 协同工作的。

ES6 & Babel

在理解这个例子的时候，你需要去了解一下 ES6 的新特性，不然的话可能理解其来会比较困难。推荐一个办法，ES6 的大概这个肯定是要去过一遍的，这个没跑，过一遍之后估计是一大半都忘记了，这个没关系，打开 babel 的官网，那里有一个 在线实验 的功能，把看不懂的代码粘贴过去，大部分其实都可以理解了。

tip：babel 是一个 JavaScript 的编译器，可以将 ES6 甚至是 ES7 的代码编译成 ES5 的代码。

entry —— index.js

在这个例子里面，/app/index.js 是整个应用的入口文件。

看到这么多的 import 其实是挺晕的，我们来一个一个的看这些 import 到底都是做什么用的。

  • babel-polyfill 它支持一些 ES6 的新特性，Object.assign() Promises Generators
  • React 使我们构建组件的框架
  • react-dom 我们从它里面导出 render 方法把根组件渲染成 DOM
  • browserHistory 这个对象是从 React-router 导出的，它可以让我们操作路由
  • syncHistoryWithStore 这个方法来自 react-router-redux，它可以同步浏览器的 history 和 redux 的 store。每当 router 变更的时候也就相当于向 store 发送一个 action
  • AppContainer 这是一个让 react 可以热加载的容器，所谓热加载就是你更改了代码不用刷新就可以在浏览器端实时的展示出来
  • configureStore 是一个非常有用的方法，可以让我们配置 store
  • Root 是最顶级的 react 组件，相当于是所有组件的入口
  • styling 样式这里也是直接 import 进来的。webpack 把样式打包进了 JavaScript 文件里面，这样可以让样式也更加模块化

就这样，我们通过 index.js，把所有的需要的东西都加载进来了。

当所有东西都加载完成之后，我们就用了一个 configureStore() 方法来配置 redux 的 store。等下写到 configureStore.js 的时候我们再详细看看。

接下来，我们创建了一个叫 history 的变量，这个 history 包含着和 store 同步浏览器 history，每当浏览器 history 变化的时候，它就会通过 router，dispatch actions 给 store。

最后，这里还有一个判断。

如果是在开发环境，就使用 热加载，每当我们更改并且保存组件里面的代码的时候，就会自动刷新。

!!!有个问题是需要格外注意的：

如果你是直接 clone 作者 github 上面的代码，你运行 node server，是没问题的，可以跑起来；
但是当你运行 npm run build 把它编译打包想在本地运行的时候，你打开 index.html 就会发现页面什么东西都没有
其实这就是 history 的锅，因为 browserHistory 是需要服务器环境的，但是你在本地没有配服务器，所以就跑步起来，解决问题的方法很简单，就是将 browserHistory 换成 hashHistory，应用在本地也可以跑了。

  • createStore 的作用就跟它的名字一样，创建一个 store，它的第一个参数 reducer，第二个参数是 initial state，第三个参数是一个扩展
  • rootReducer 是我们的 reducer，这个后面我们详细来讲。现在我们只要知道 reducer 是一个纯函数，传递旧的 state 和一个 action 进去，它就会返回一个新的 state

Reducers

现在我们已经知道了 store 是怎么创建的，现在就来看看它里面的 Reducers，打开 /app/reducers/index.js。

  • routerReducer 是 react-router-redux 创建的，它是用来处理路由变化的 reducer。
  • combineReducers 组合两个或者多个 reducer 并且把它们合并到一个 state 树下。这样表达可能有点绕，我们可以这样来理解，reduex 让全局只有一个总的 state，那就是顶层的 state，combineReducers 里面的每个 reducer 也会返回 state，把它们合并到一个 state 树下的意思就是说，将那些 reducer 返回的 state 变成顶层 state 的一个属性，属性名就是 reducer 的名字。
  • actions 这个是我们自己定义的，稍后再看看

Root

我们在 index.js 那里就说过， Root 是所有组件的入口。其实看它所在的文件路径也应该知道，它在 /app/containers/ 里面，containers 就是整个组件所有的子组件都会被 contain 在它里面。

Root 接收 redux 的 store 和 history，把他们作为 props 保存起来。

当你打开 app/containers/Root.js 的时候，你会发现它跟 configureStore.js 那个文件一样，有个对于环境的判断。

现在，我们来看下开发模式的版本。

我们先来看看 import 进来的都是什么东西：

  • 我们从 react 里面 import 了 Component 和 PropTypes 这两个对象进来，其中 Component 是创建 component 的时候用到的，PropTypes 是检测 props 类型的时候用到的
  • provider 来自 react-redux，它可以订阅包裹在它里面的所有组件
  • Router 这块比较大，建议还是去查阅文档，简单的来说，这个 Router 是用来配置我们整个程序的路由的
  • devtools 这个就是我们在开发环境下运行程序的时候在右边看到的那一个很酷的监测面板

总的来说，Root 所做的就是包裹了一个 Provider 组件。

App

现在我们来看看 /app/components/App.js

你也许会有疑问，为什么我们的应用要分为 /components 和 /containers，Redux 文档会给你答案。
简单来说，在 /components 目录下的都是 “dumb(哑)” 组件，为什么叫做 dumb 组件呢？因为它们不直接接触 redux 的 store，它们只接收父组件传递给它们的属性。只有 /containers 连接到 redux 的 store 并且整个应用的 state 都通过它来管理。

下面说回 App 组件，App 组件是一个随处可见的组件，类似于 tab 选项卡的包裹容器，姑且就这样理解吧。
